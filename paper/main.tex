\begin{abstract}
Real-world travel requests are often global, multi-modal, and interactive: a user may ask for a trip that combines local access, long-distance transportation, ground transfers, and clarification of underspecified endpoints. We present \sys, an LLM-based system that uses an LLM for intent extraction, ambiguity clarification, and instruction decomposition, while a routing backend generates feasible routes from OpenStreetMap, OpenFlights, Overpass, and mode-specific graph search. \sys presents segment-level route alternatives in an interactive interface, allowing users to inspect, compare, and revise walking, driving, public-transit, flight, ferry, and transfer options. We also introduce \bench, a 100-example benchmark for evaluating system-level success in global multi-modal route planning; the benchmark retains optional route-playback annotations for future instruction-following studies, but the demo evaluation focuses on whether the system resolves user requests and returns feasible route plans. Across \bench and 1,723 external route-request instances derived from MultiWOZ 2.4, Schema-Guided Dialogue, and ATIS, we evaluate \sys, with the best LLM-backed configuration achieving 86\% system success on \bench and 95.2\% on the external set.




\end{abstract}

\section{Introduction}

Real-world travel planning is rarely a single-mode shortest-path problem~\citep{bast-etal-2016-route-planning}. A user may ask for a trip that combines walking to a station, taking public transit to an airport, flying to another city, and choosing between a taxi, ferry, or walking route for the final segment. The same user may also express personal preferences, such as avoiding driving, preferring a scenic ferry, minimizing transfers, or asking for a cheaper public-transport option. As AI assistants become a more natural interface for everyday services~\citep{rastogi-etal-2020-schema}, navigation systems increasingly need to recover structured travel intent from informal, underspecified language rather than from pre-filled forms~\citep{hemphill-etal-1990-atis}. A practical system must therefore (i) extract origins, destinations, and constraints from free-form user requests; (ii) ask targeted clarification questions when endpoints are missing or ambiguous; (iii) determine feasible transportation modes from map and transport data rather than language priors alone; and (iv) expose alternative segment-level choices to the user, since the ``best'' route is often subjective.


LLMs have advanced rapidly as general-purpose language interfaces: scaling improves few-shot task adaptation~\citep{NEURIPS2020_1457c0d6}, chain-of-thought prompting improves multi-step reasoning~\citep{NEURIPS2022_9d560961}, and recent work explicitly studies their spatial-reasoning and path-generation capabilities~\citep{rizvi-etal-2024-sparc}. However, existing route-planning interfaces and LLM assistants still leave a gap between natural language travel intent and grounded multi-modal routes. Conventional map interfaces are strong once the origin, destination, and mode are specified, but they usually assume that the user has already resolved ambiguous places, decomposed a complex trip into stages, and chosen which transportation modes to compare. Recent language-agent travel-planning benchmarks similarly show that LLM agents struggle with tool use and constraint tracking in realistic travel tasks~\citep{pmlr-v235-xie24j}. Pure LLM planning, by contrast, can produce fluent itineraries but cannot by itself guarantee that an airport transfer, ferry leg, road segment, or public-transit option is supported by external mobility evidence.

To address these requirements, we introduce \sys, an LLM-based, environment-driven demo for global route planning. \sys uses the LLM for language-facing operations, including intent parsing, ambiguity clarification, and high-level route explanation, while route feasibility and travel options are determined by a routing environment. For local and access segments, the environment constructs route candidates from OSRM road routes, OSM/Overpass stops and route relations, and walking transfers. It returns a driving option when the road network supports one, preserves a pure walking option as a mandatory reference, and exposes alternative routes according to travel time and user preferences. For long-range segments, the environment connects local access and egress segments with verified long-distance connectivity: OpenFlights provides air-route connectivity, while ferry/maritime legs are grounded by OSM/Overpass ferry-terminal evidence and estimated maritime phases rather than a dedicated ferry timetable database. The same route-option layer also supports bus, train, and tram alternatives for access and egress when OSM/Overpass evidence or estimated transit edges are available.

This paper makes three contributions:
\begin{enumerate}[leftmargin=*]
\item We introduce \sys, an environment-driven global multi-modal route-planning demo that uses LLMs for language understanding and clarification while grounding route feasibility and travel options in an external mobility environment.
\item We construct \bench, a benchmark with an annotation protocol for evaluating global multi-modal navigation requests, covering intent parsing, ambiguity clarification, environment-grounded route generation, and optional route-playback annotations.
\item We provide a focused system-level evaluation of the demo on \bench, plus an external origin--destination parsing evaluation over MultiWOZ 2.4, Schema-Guided Dialogue, and ATIS.
\end{enumerate}

\section{\bench: Benchmark Design}

\bench is designed to evaluate global multi-modal navigation beyond pedestrian-only street-view navigation. It contains 100 instructions spanning local, regional, and global routes. These instructions cover a range of navigation settings, from short urban trips to long-distance journeys that require combining local access, inter-city or cross-region travel, and local egress. Each benchmark example follows the schema shown in Figure~\ref{fig:schema}, which records the user instruction, parsed intent, clarification status, route segments, available transportation modes, and optional route-playback information.



\subsection{Data Collection Principles}

We constructed \bench around four design principles. First, the benchmark should cover multiple spatial scales rather than only local pedestrian navigation. Examples therefore include same-city routes, airport or station access routes, cross-region routes, and global routes involving long-distance transportation. Second, the benchmark should include multiple transportation modes and mode transitions. Each route may involve walking, driving, public transport, flights, ferries, or combinations of these modes, depending on the geographic context and available route evidence. Third, each example must be grounded in verifiable geographic and transportation evidence. Annotators record the intended origin and destination for each instruction, and the route-generation module links these endpoints to available map and transportation data. Local movement is grounded in geocoding results and OSRM route geometry, while public-transport and long-distance segments are supported by OSM/Overpass evidence, OpenFlights connectivity, or another verified source when available. Examples that cannot be fully verified from the available data are not included in \bench.
Fourth, the benchmark should preserve route-playback annotations as a secondary resource. These annotations are not needed to judge demo success, but they make the collected routes reusable for later work on embodied instruction following.

\subsection{Data Collection and Annotation Workflow}

We first collect a geographically diverse set of city and place seeds across macro-regions, including North America, Europe, Asia, Oceania, Africa, South America, and the Middle East. For each city, we select representative points of interest such as hotels, landmarks, conference venues, rail stations, airports, ports, and ferry terminals. For each selected place, we record its canonical place name, geocoding query, coordinates, place type, and relevant map metadata. These fields are used to verify that the origin, destination, and intermediate transportation hubs correspond to real geographic locations before the example is included in the benchmark.

Given these seed places, we generate origin--destination pairs under different route settings. Local examples pair two places within the same city. Access examples connect an arbitrary point of interest to a transportation hub such as an airport, rail station, or port. Regional and global examples pair places across different cities or macro-regions, requiring the system to combine local access, long-distance transportation, and local egress. We also construct ambiguous or under-specified instructions, such as missing origins, missing destinations, ambiguous place names, ambiguous airport references, and underspecified transport preferences. These examples are used to evaluate whether a model can ask an appropriate clarification question before route generation.

For each instruction, the mobility environment produces candidate route annotations. Local route options are generated from OSRM walking and driving routes and from available OSM/Overpass public-transport evidence. For long-distance routes, the environment identifies relevant transportation hubs and checks air connectivity against OpenFlights when available. Each route annotation records the expected segment order, allowed transportation modes, evidence source, and known route-generation requirements, such as retaining a pure walking option for local segments or including both access and egress legs for global trips.

Human annotators then review the generated examples. They verify endpoint correctness, correct or reject invalid transfer points, check whether the segment order is plausible, and mark each leg with its evidence status. For clarification examples, annotators label whether a follow-up question is required and assign a gold clarification type. For route-option examples, annotators verify whether the returned mode choices are meaningful alternatives rather than duplicates of the same route. To make the benchmark reusable beyond the demo, annotators also create optional mode-specific procedural notes for local self-controlled movement. For driving routes, annotators are asked to imagine themselves as a passenger-side navigator guiding the driver. They write clear and actionable notes based on observable road context, such as ``turn right at the second traffic light'' or ``turn left at the first traffic light after the Starbucks.'' For walking routes, we follow the annotation principle of street-view navigation benchmarks such as Touchdown, where instructions are grounded in visible landmarks and pedestrian-scale movement~\citep{Chen_2019_CVPR}. A second annotator validates the final record, including the intent labels, clarification labels, route segments, evidence fields, and optional playback annotations.

\paragraph{Local walking/driving.}
This category covers same-city navigation, including walking and driving directions from an origin to a nearby destination, such as from a campus building to a city landmark. Each route is paired with optional procedural notes written in the style of a human passenger or co-driver, such as ``continue on Broadway,'' ``turn left at the third traffic light,'' or ``keep the station on your right.'' The main demo evaluation uses these examples to test endpoint resolution and feasible route construction.

\paragraph{Global access and hybrid.}
This category covers trips that leave the immediate local street network, including routes from arbitrary points of interest to airports, ports, or stations as well as access + flight/ferry + egress journeys across cities or macro-regions. A typical example may require walking or driving to a hub, taking a flight or ferry to another city, and then continuing by driving, walking, or public transport to the final destination. For non-self-controlled modes such as flights, trains, and ferries, the playback module records phase transitions once the route reaches the departure hub.

\paragraph{Ambiguity annotations.}
For clarification evaluation, each example is annotated with whether a follow-up question is required. When clarification is needed, we further annotate the ambiguity type, such as missing origin, missing destination, ambiguous place name, ambiguous airport, or underspecified transportation preference. These annotations support evaluation of whether a model can detect unresolved intent and ask an appropriate clarification question before route generation.

\subsection{Annotation Protocol}

We introduce a semi-automatic annotation workflow.
First, sample origin--destination pairs from real map queries. Second, use \sys to generate candidate route segments and evidence, including OSRM road geometry, OpenFlights connectivity, OSM/Overpass stops, and verified transit or ferry evidence when available. Third, human annotators verify the endpoints, choose or correct target transfer points, and reject impossible transfers; examples that cannot be fully grounded in the available evidence are excluded before final benchmark inclusion. Fourth, for local walking/driving legs, annotators may write procedural notes from a co-driver perspective, e.g., ``at the third traffic light, turn left,'' rather than asking the LLM to navigate directly from a raw address. For flight and ferry legs, annotators label phase transitions such as board, takeoff, cruise, land, depart, and dock. Finally, a second annotator validates the instruction, route evidence, clarification label, and optional playback record.

This protocol intentionally differs from classic VLN annotation. Instead of only asking a worker to describe a walking path, annotators validate a complete route-planning record: language intent, endpoint grounding, route evidence, mode alternatives, and optional playback notes. This supports the paper's central claim: a global navigation demo must resolve and ground a route before any downstream instruction-following behavior is meaningful.

\section{Experiments}

\bench is evaluated with a system-level demo experiment: whether the full pipeline resolves the user request, produces a feasible route, and handles clarification when needed. We also run a full-pool external origin--destination (OD) evaluation to isolate the language-facing parser and clarification loop on existing dialogue and travel-query datasets. We treat route construction as environment infrastructure rather than as an LLM planning task: given verified endpoints, the route is generated by map and transportation data, not by the model. Accordingly, we do not include a separate experiment for route-option generation; route options are checked during annotation and illustrated in case studies. Appendix~\ref{sec:follower_appendix} reports a secondary instruction-follower analysis for readers interested in the reusable benchmark annotations, but it is not a main demo claim.

\subsection{Experiment Setup}

\paragraph{Implementation Details.}
The \bench system-level experiment is evaluated on the full set of 100 instructions; no held-out partition is used because no model is trained on the benchmark. Each example includes verified route evidence, gold origin and destination entities, ask/no-ask clarification labels, and ambiguity types. We evaluate recent closed-source LLMs including GPT-5.5, GPT-5 mini, GPT-4.1, GPT-4.1 mini, Claude Sonnet 4.6, and Gemini 2.5 Pro. The OpenAI mini models provide lower-latency and lower-cost configurations for comparison against their full-size counterparts under the same \sys pipeline. Unless otherwise specified, decoding uses temperature 0, and each example is run once.


\paragraph{System Success Metrics.}
System success rate (SSR) is the percentage of benchmark examples for which \sys end-to-end resolves the user request and returns a feasible route after any required clarification. We also report component success rates to localize failures: intent resolution measures whether the resolved origin, destination, and constraints match the annotated canonical entities after geocoding; route validity measures whether the mobility environment returns a verified feasible route for the resolved intent; clarification resolution measures whether ambiguous examples receive an appropriate follow-up question and are correctly resolved before routing.



\subsection{System-Level Success}

\paragraph{Purpose.}
This experiment evaluates whether \sys succeeds as a complete route-planning demo. A successful run requires resolving the user intent (including clarification when needed), grounding the endpoints in map data, and returning a verified feasible route from the mobility environment.

\paragraph{Evaluation.}
For intent resolution, an origin or destination is correct if it matches the annotated canonical entity after geocoding; when only coordinates are available, we use distance thresholds by entity type: 100 meters for POIs and buildings, 1 km for airports, stations, and ports, and the annotated administrative area or 10 km from the city center for city-level endpoints. Route validity requires that every leg in the returned plan is supported by the annotated evidence policy used during benchmark construction. For ambiguous examples, clarification resolution is counted as success only when the system asks an appropriate follow-up question and the resolved intent matches the gold annotation.
Table~\ref{tab:system} reports system-level success for the rule baseline and LLM-backed \sys configurations.

\subsection{External OD Parser Evaluation}

\paragraph{Purpose.}
The system-level experiment above evaluates \sys on the collected \bench instructions, where route validity is checked against the benchmark's verified route evidence. To test the deployable language-facing component at larger scale, we additionally evaluate the parser and clarification loop on external OD-like examples derived from existing datasets: MultiWOZ 2.4~\citep{ye-etal-2022-multiwoz}, Schema-Guided Dialogue (SGD)~\citep{rastogi-etal-2020-schema}, and ATIS~\citep{hemphill-etal-1990-atis}. This experiment intentionally excludes route validity because the external datasets provide origins and destinations but not a verified \sys mobility graph; it therefore measures whether the demo system can resolve the user's OD intent, not whether an externally supplied road or transit network is correct.

\paragraph{Data and protocol.}
We evaluate the deployed \texttt{gpt-5-mini} parser on the full eligible external OD pool: 381 examples from MultiWOZ 2.4, 733 from SGD, and 609 from ATIS (1,723 total). For MultiWOZ and SGD, explicit examples are constructed from original turns that contain both origin and destination information, while clarification examples are constructed from original incomplete user turns followed by the dataset's next user answer. ATIS contributes single-turn explicit flight queries only. The \texttt{expected\_clarification} label is not a native field in these datasets; it is deterministically derived from native dialogue state, dialogue acts, or BIO slot annotations by checking whether the first user-visible request already contains both OD endpoints. During evaluation, the system first receives only the initial user request. If and only if it asks the correct type of clarification question (missing origin vs. missing destination), the evaluator supplies the original next user answer before scoring the final parse. No LLM judge is used.

\paragraph{Metrics.}
Ask/Act measures whether the system correctly decides to answer directly or ask a clarification. Target measures whether clarification questions ask for the missing endpoint; it is defined only for native clarification examples. Origin and destination accuracy compare the final parsed endpoints against the gold strings after lightweight normalization. Strict success requires all applicable checks to pass: the correct ask/act decision, the correct clarification target when clarification is expected, a final non-ambiguous parse, and correct origin and destination.

\section{Conclusion}

We introduced \sys, an environment-driven global multi-modal route-planning demo that moves beyond city-scale, single-mode navigation requests.
By separating LLM language understanding from route feasibility, \sys avoids treating an LLM as an unconstrained route planner. The main evaluation shows how often the deployed system resolves user intent, handles clarification, and returns a feasible route, while the external OD evaluation stress-tests the language-facing parser on existing dialogue and travel-query datasets. Together, the system and benchmark aim to support a new class of interactive global navigation tools that can reason over walking, driving, public transit, flights, ferries, and user preferences while remaining grounded in external mobility evidence.

\section*{Limitations}

\sys does not download a complete global map.
It queries route and map data on demand and caches results.
This makes the demo lightweight but dependent on API availability and coverage. OpenFlights only provides connectivity, not real-time schedules or ticket availability. Public-transit, ferry, and rail coverage depends on available public evidence; when schedules are unavailable, these modes are evaluated for route evidence and mode-chain validity rather than timetable duration. The optional playback annotations in \bench are useful for later instruction-following work, but the present demo paper does not claim a full embodied-navigation evaluation.

\section{Prompt and Evaluation Details}
\label{sec:prompts}

\paragraph{Intent parsing prompt.}
The parser is constrained to extract origin and destination only. It is instructed not to plan routes or choose transport modes, and to set \texttt{needs\_clarification=true} only when an endpoint is genuinely missing or ambiguous. The full prompt is implemented in \texttt{globe\_nav/llm\_parser.py}.

\paragraph{Clarification prompt.}
Clarification is implemented as a one-turn loop over the same parser. The system stores the category and text of the question, appends the user's answer to a clarification history, and reparses the original instruction with that history. The evaluation supplies a gold answer only when the system asks for the gold missing endpoint.

\paragraph{Follower decision prompt.}
The optional follower prompt verbalizes the current segment, mode, heading, route cue, procedural reminder, and mode-specific action space. It is used only in the appendix experiment and is implemented in \texttt{globe\_nav/follower/verbalizer.py}.

\section{Optional Instruction-Follower Evaluation}
\label{sec:follower_appendix}

This appendix records an auxiliary use of \bench: evaluating whether an LLM decision agent can follow the optional playback annotations. This experiment is not a main demo result, because the demo contribution is the route-planning system and user interaction rather than embodied action selection.

\paragraph{Metrics.}
Task completion (TC) is the percentage of episodes in which the agent reaches the final goal within the step budget. Shortest-path distance (SPD) measures the remaining shortest-path distance from the agent's final state to the goal, so lower is better. We report normalized Dynamic Time Warping (nDTW) because it compares a predicted trajectory with a reference trajectory while tolerating small local deviations~\citep{ilharco-etal-2019-ndtw}. Correct-action distance (CAD) measures the cumulative distance traveled while the agent's selected actions match the oracle action at annotated decision points; higher values indicate that the agent follows the route correctly for longer.

\paragraph{Protocol.}
The simulator sanity check verifies that the route can be instantiated and that the oracle policy reaches the goal. The LLM decision agent receives the current location, heading, mode, route context, available actions, and procedural notes at each step, then selects one action. For non-self-controlled modes such as flights, ferries, and trains, automatic vehicle movement is not counted as agent transition accuracy. If the action space includes explicit board, disembark, or continue-egress actions, those actions are counted under segment success.
